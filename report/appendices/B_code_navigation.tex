\chapter{Code Navigation}

\section{How to Read the Codebase}

The implementation is organized as a reusable Python package plus a small set of command-line scripts.
For a reader who wants to inspect the code without reading every file line by line, the most useful path is:

\begin{enumerate}
    \item start from \path{scripts/run_experiment.py}, which defines the command-line interface for the main experiment;
    \item inspect \path{src/nn_option_pricing/config.py}, where experiment parameters are represented as immutable configuration objects;
    \item follow \path{src/nn_option_pricing/pipeline.py}, which connects the main experimental stages;
    \item inspect the specialized modules only when needed.
\end{enumerate}
This gives a top-down view of the project before moving into the individual numerical and machine-learning components.

The repository contains separate scripts for the main experiment and for additional benchmarks.

\section{GitHub Code Links}

When the report is read together with the GitHub repository, code links can be used to inspect specific implementation details.
For stable references, it is preferable to link to a fixed commit rather than to a moving branch.
This avoids broken or misleading line references if the code changes later.

The most useful targets for direct GitHub links are:

\begin{itemize}
    \item the analytical pricing implementation in \path{black_scholes.py};
    \item the synthetic dataset generator in \path{dataset.py};
    \item the training loop in \path{training.py};
    \item the end-to-end pipeline in \path{pipeline.py};
    \item the Monte Carlo benchmark in \path{monte_carlo.py};
    \item the SVR and noisy-target extensions in \path{svr.py}, \path{noise.py}, and the noisy experiment modules.
\end{itemize}
These links are not a substitute for the report, but they make it easier for a reader to verify how the mathematical and experimental choices are implemented in code.
